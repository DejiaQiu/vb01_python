# 功能需求单

## 目标
- 将钢丝绳松动专项诊断从依赖单梯绝对阈值的规则，改造成高精度优先的候选故障筛查算法。
- 将橡胶圈硬化专项诊断接入统一候选故障筛查链路，并使判定逻辑更贴合“曳引机本体安装振动传感器”场景下的支撑刚度/阻尼变化机理。
- 在更换电梯、安装位置或传感器幅值尺度变化时，降低分数漂移和误报风险，允许一定漏报但尽量不把正常工况报成异常。
- 为 Dify 展示链路补齐波形图生成能力，使“读取 CSV -> 绘制波形图 -> 综合研判 -> 生成报告”可以通过后端 API 和可导入 workflow 串起来。
- 增加“定时批诊断 + 最新状态查询”链路，使项目可以每天定时跑几次离线 CSV，用钢丝绳松动/橡胶圈硬化候选筛查算法生成最新状态，而不是继续以旧实时规则算法作为在线主判。

## 用户场景
- 设备诊断人员会对单个或批量振动 CSV 执行钢丝绳松动专项诊断。
- 维保或平台系统每天会定时采集几批 CSV，并需要生成“当前状态 / 当前风险 / 最近候选故障”的在线回复结果。
- 在同一套算法需要迁移到不同电梯或不同采集批次时触发本需求。
- 当没有足够健康基线时，算法应回退到仅依赖无量纲自归一化特征的保守模式，避免直接失效。
- 算法输出的是“疑似钢丝绳松动候选”，用于辅助维保筛查，不要求精确到松动几根钢丝绳。
- 算法输出的是“疑似橡胶圈硬化候选”，用于辅助维保筛查，不要求直接推断具体哪一个橡胶件失效。
- 在线问答不直接访问旧实时监控故障类型，而是查询最近一次定时批诊断结果。

## 范围
- 包含：
  - 修改钢丝绳松动专项规则算法，使其优先使用健康基线做 robust 归一化评分，并将判断口径收敛到高精度候选故障筛查。
  - 新增橡胶圈硬化专项规则算法，使用健康基线对竖向响应、方向性耦合、阻尼退化代理量做相对评分，并以保守单窗确认形式输出高置信候选。
  - 搭建 `report` 链路的统一候选故障筛查骨架：各故障脚本独立打分，由总控统一输出高置信候选、观察候选和原始排序结果。
  - 新增波形图生成 API，输出用于 Dify 报告展示的加速度、角速度和加速度合成幅值波形图。
  - 新增或更新可导入的 Dify workflow DSL，使其通过 HTTP 节点串联规则诊断、波形图、维保包和综合报告接口。
  - 新增定时批诊断入口，复用 `report/fault_algorithms` 的候选故障筛查算法，对最近一批 CSV 生成最新状态文件和历史记录文件。
  - 新增最新状态查询 API，供 Dify 在线回复读取最近一次定时批诊断结果。
  - 将在线状态问答集成到现有 Dify workflow 中：未上传 CSV 时查询最新状态，上传 CSV 时生成离线诊断报告。
  - 保留现有 JSON 输出结构，并补充基线、结构证据、门控状态等可解释字段。
  - 更新时间序列确认逻辑，使其在低质量窗口存在时仍保持保守且稳定。
  - 新增或修改自动化测试，覆盖泛化性、正常抑制、回退逻辑、统一候选输出与连续确认。
- 不包含：
  - 重写底层串口采集流程、训练流程或旧实时监控进程。
  - 引入新的第三方依赖。

## 接口与输入输出
- 输入：
  - 保持现有 `detect(features: dict) -> dict` 入口兼容。
  - 允许 `features` 中可选传入 `baseline` 字段，内容为钢丝绳或橡胶圈硬化相关特征的健康基线统计量。
  - 保持现有 CLI 输入 CSV 方式不变；时间序列确认脚本和统一故障筛查入口可选接受基线文件或健康样本目录。
  - 新增波形图 API 输入，兼容 `csv_path`、`csv_text`、`rows` 三种方式。
  - 新增定时批诊断 CLI/API 输入，至少支持 `input_dir`、`csv_paths`、`max_files`、`baseline_json`、`baseline_dir`、状态输出路径。
- 输出：
  - 保持现有结果 JSON 的 `fault_type`、`score`、`level`、`triggered`、`quality_factor`、`reasons`、`feature_snapshot` 字段兼容。
  - 在 `reasons` 中增加 robust 分量、基线状态、回退模式、结构型放行与门控证据。
  - 统一故障筛查入口除原始结果外，还应输出高置信候选列表、观察候选列表和筛查状态。
  - 时间序列确认输出继续返回单窗分数和连续确认结果。
  - 波形图 API 至少输出加速度三轴图、角速度三轴图、加速度合成幅值图以及可直接用于 Markdown 展示的图像数据。
- 定时批诊断输出至少包含：最近一次状态、最新候选故障、最近一批窗口摘要、风险分数、风险等级、状态文件路径。
- 最新状态查询 API 至少返回最近一次批诊断结果和状态文件路径。
- 定时批诊断写出的 `latest_status.json` 应额外包含后端生成的 `report_markdown_draft`，供 Dify 在线状态链路直接展示。
- 最新状态查询 API 应支持通过 `elevator_id` + 状态根目录自动定位对应电梯的 `latest_status.json`，以支持多电梯在线状态查询。

## 数据结构
- 新增钢丝绳诊断所需的健康基线统计字段，至少包含相关特征的 `median` 和 `scale`。
- 新增橡胶圈硬化诊断所需的按轴统计字段与相关健康基线统计字段，至少包含 `median` 和 `scale`。
- 新增波形图数据结构，至少包含图名、SVG 或 data URI、Markdown 表达和数据摘要。
- 新增定时批诊断状态结构，至少包含 `generated_at_ms`、`status`、`top_candidate`、`risk`、`history`。
- 其余结构不新增独立 schema 文件。

## 业务规则
- 钢丝绳松动诊断应优先使用健康基线对少量核心特征做 robust 归一化，核心特征收敛为 `lateral_ratio`、`lat_dom_freq_hz`、`lat_low_band_ratio` 以及耦合聚合量（`ag_corr/gx_ax_corr/gy_ay_corr`）。
- 钢丝绳通用异常辅助特征只保留 `a_rms_ac`、`a_p2p`、`g_std`；冲击类特征作为惩罚项。
- 当提供健康基线时，算法应尽量对整体幅值的等比例缩放保持近似不变。
- 当未提供健康基线时，算法应回退到仅依赖无量纲比值和相关性特征的保守模式，不得因缺少基线而报错。
- 触发逻辑应继续保守，优先保证正常工况不触发；允许漏报，但不能为了提高召回率显著增加误报。
- 单窗口除“摆动增强”外，还允许“横向失衡 + 耦合增强 + 低频成立 + 非冲击”的结构型证据通道，但不得再依赖大量分支阈值堆叠。
- 基线构建应尽量排除样本点过少的低质量窗口，避免健康画像被异常窗或空窗污染。
- 时间序列确认在遇到极低质量窗口时可跳过该窗，但不得因此把单点噪声放大成连续故障。
- 统一故障筛查总控不得直接把原始最高分等同于最终故障结论；应优先输出高置信候选故障，并保留原始排序供复盘。
- 当没有任何高置信候选时，总控应保持 `normal` 或 `watch_only` 状态，而不是强行给出唯一故障结论。
- 总体流程应明确分层为：`健康基线 -> 通用异常门 -> 类型判断（钢丝绳 / 橡胶圈）-> 最终决策层`，不得把单个专项脚本直接当作最终用户结论来源。
- 通用异常门只判断相对健康基线是否存在明显偏移，不直接输出故障类型；故障类型应由后续专项规则判断。
- 钢丝绳专项规则应主要依赖横向比例、横向低频特征和加速度/角速度耦合聚合量等钢丝绳核心特征；共同异常特征只允许作为弱辅助。
- 钢丝绳专项应保持规则算法实现，不保留 centroid 模型训练、模型加载或规则-模型分数融合链路；跨梯验证只用于评估规则泛化性。
- 橡胶圈专项规则应主要依赖 `energy_z_over_xy`、`az_p2p`、`az_cv`、`az_jerk_rms` 和轴间耦合变化等少量核心特征；共同异常特征只允许作为弱辅助。
- 共享异常特征只能支持 `watch_only` 或决策解释，不能单独把样本提升为某一类高置信候选。
- 最终决策层应输出统一主结论，并保留 `rope_primary`、`rubber_primary`、`system_abnormality` 和 `primary_issue` 等结构化字段供报告和接口消费。
- 当通用异常门判断为异常，但钢丝绳与橡胶圈专项规则都没有形成足够的类型专属性时，最终决策层应输出 `unknown` 作为观察级兜底主结论，而不是强行二选一。
- 波形图 API 应优先复用已有 CSV 解析和 `is_new_frame` 过滤逻辑，避免把补点直接绘成主波形。
- 诊断报告接口在请求显式开启时，应可自动附带波形图结果，供 Dify 报告展示使用。
- 橡胶圈硬化诊断应贴合“曳引机本体安装传感器”的观测条件，优先比较健康基线上的竖向响应增强、阻尼退化代理量增强、轴间耦合结构变化，而不是只依赖总振动绝对幅值。
- 橡胶圈硬化诊断中，竖向统计量（如 `az_std`、`az_rms_ac`、`az_p2p`、`az_cv`）、方向性代理量（如 `energy_z_over_xy`）、轴间相关结构（如 `corr_xy`、`corr_xz`、`corr_yz`）和瞬态/阻尼代理量（如 `az_jerk_rms`、`mag_std`、`mag_cv`）应参与综合评分。
- 橡胶圈硬化诊断应对冲击型样本保持抑制，避免把碰撞、急停或明显尖峰冲击误报成硬化。
- 当缺少健康基线时，橡胶圈硬化诊断可退回到更保守的方向性/竖向响应规则，但不应因为没有硬化模板样本就完全不可用。
- 定时批诊断应优先复用离线候选故障筛查算法，不应继续把旧实时 `fault_types.py` 里的通用故障规则当作在线主判来源。
- 定时批诊断应支持从目录中选取最近若干个 CSV 进行批处理，并把最近一次结果写入稳定路径，供在线状态查询接口读取。
- 最新状态中的风险分数应综合最近一次候选分数、最近多次重复出现的比例、分数趋势和数据质量，而不是沿用旧实时风险分数。
- 当最近一次结果为 `normal` 但此前多次存在候选时，风险应允许回落，但不能立即清零。
- 集成后的 Dify workflow 在未上传 CSV 时应仅查询最新状态接口，在上传 CSV 时应走离线诊断报告链路。
- 集成后的 Dify workflow 在未上传 CSV 且用户问题中包含电梯编号时，应优先按该 `elevator_id` 查询对应电梯的最新状态。

## 验收标准
- 在提供健康基线时，对同一物理模式做近似等比例幅值缩放，诊断结论和分数应保持基本一致，不得因尺度变化从 `triggered=true` 变为 `false`。
- 在健康或近健康样本上，钢丝绳脚本应保持保守，不得因正常波动频繁触发告警。
- 在没有健康基线时，算法仍能返回结构完整的结果，并对明显冲击型样本保持抑制，不误报为钢丝绳松动。
- 在健康或近健康样本上，橡胶圈硬化脚本应保持保守，不得因正常工况变化或单次尖峰冲击频繁触发告警。
- 在提供健康基线时，明显的“竖向响应增强 + 耦合结构变化 + 非冲击”样本可进入橡胶圈硬化候选区。
- 时间序列确认脚本在新评分下仍能输出连续确认结果，且兼容原有批量 CSV 输入方式。
- 统一故障筛查入口在没有高置信候选时，应输出空候选列表或正常状态，不得仅凭原始最高分触发维保建议。
- 当系统存在明显基线偏移但专项类型证据不足时，统一故障筛查入口应返回 `watch_only + unknown` 的兜底结果。
- 报告、批诊断和在线状态查询应优先读取统一决策层给出的 `primary_issue`，而不是重复在展示层自行推断主结论。
- 在线状态链路应优先复用后台批诊断写出的 `report_markdown_draft`，避免在展示层重新拼接与后端不一致的报告结论。
- 波形图 API 对有效样本充足的输入应返回结构完整的图像结果，对低质量输入也应返回可解释的摘要而不是 500 错误。
- Dify workflow DSL 文件应可被 YAML 解析，并包含规则诊断、波形图和报告相关节点。
- 定时批诊断 CLI/API 对一批 CSV 执行后，应产出结构完整的最新状态 JSON，并可选追加历史 JSONL。
- 最新状态查询 API 在状态文件存在时应返回 200 和结构化结果，在文件不存在时返回可解释错误。
- 集成后的 Dify workflow DSL 应可被 YAML 解析，并在未上传 CSV 时只依赖状态查询接口完成回答。

## 测试用例
- 新增测试：提供健康基线时，同一摆动异常样本做幅值缩放后，触发结果一致且分数差异受限。
- 新增测试：提供健康基线时，接近健康画像的样本保持非触发。
- 新增测试：没有健康基线时，低频横向摆动异常可进入告警区，而冲击型样本仍被抑制。
- 新增测试：橡胶圈硬化在健康基线下，对明显竖向响应增强与耦合结构变化的样本输出候选，而正常样本保持非触发。
- 新增测试：橡胶圈硬化在没有健康基线时，保守 fallback 评分仍能区分明显硬化型样本与冲击型样本。
- 新增测试：时间序列确认逻辑在混合正常窗、低质量窗与异常窗的序列上，只在满足连续窗口数时输出确认告警。
- 新增测试：统一故障筛查入口对正常样本输出空高置信候选，对明显钢丝绳异常样本输出钢丝绳候选，对明显橡胶圈硬化样本输出橡胶圈硬化候选。
- 新增测试：当统一决策层给出 `primary_issue` 时，批诊断和报告链路应优先使用该字段，而不是回退到旧的 `top_candidate` / `watch_faults` 推断逻辑。
- 新增测试：波形图 API 对内联 rows 输入返回加速度图、角速度图和幅值图字段。
- 新增测试：诊断报告接口在 `include_waveforms=true` 时返回波形图载荷，并在 Markdown 草稿中包含波形图段落。
- 新增测试：定时批诊断对最近一批 CSV 运行后，会写出 `latest_status.json`，并在结果中返回最新候选故障和风险字段。
- 新增测试：最新状态查询 API 能读取已写出的状态文件，并返回状态摘要。
- 新增测试：定时批诊断写出的 `latest_status.json` 包含 `report_markdown_draft`，最新状态查询 API 原样返回该字段。

## 非目标
- 不做基于监督学习的新模型训练。
- 不保留钢丝绳专项的 centroid 模型兼容链路。
- 不把旧实时故障规则继续当作新在线状态的主判链路。
- 不恢复旧版依赖硬化模板样本的实验脚本。
- 不将本次输出定义为“精确诊断松动几根钢丝绳”。
- 不重新定义全项目统一告警等级。
